## SECTION C — Build-Tooling Architecture Analysis

### Build-tooling domain (general WordPress architectural
knowledge)

WordPress build-tooling architecture (Phase 7-aware
analysis):

#### C.1 — @wordpress/scripts (wp-scripts) infrastructure

- Provides webpack + babel + Jest pre-configured for
  WordPress block authoring
- `npm run build` produces production assets in `build/`
  directory
- `npm run start` provides hot reload for editor development
- Auto-detects entry points from block.json files
- Includes WordPress-specific webpack plugins:
  - `@wordpress/dependency-extraction-webpack-plugin`
    (extracts wp-* dependencies into runtime-loaded
    handles)
  - `@wordpress/i18n-loader-webpack-plugin` (translation
    extraction)
  - `@wordpress/babel-preset-default` (JSX + TypeScript
    + WordPress conventions)

#### C.2 — block.json compile-time processing

- Source `src/block.json` declares block metadata
- Build process copies block.json to `build/` (potentially
  with transformations)
- Asset URLs in block.json (`file:./index.js` etc.) resolve
  to built file paths
- WP 6.8+ adds `wp_register_block_metadata_collection()`
  with `block-manifest.json` for batch registration —
  build-tooling generates this manifest

#### C.3 — Asset bundling pipeline

- `editorScript`, `script`, `viewScript` bundle generation
- CSS processing (`editorStyle`, `style` via PostCSS)
- Asset hash generation for cache busting
- Source map generation
- Tree shaking + minification

#### C.4 — Translation file generation

- POT extraction from PHP source via `wp i18n make-pot`
- JSON generation for JS scripts via `wp i18n make-json`
- Build-tooling integrates with translation file lifecycle

#### C.5 — Configuration declarations

- `package.json` (npm dependencies + scripts)
- `webpack.config.js` (build customization)
- `block.json` (per-block metadata)
- `.eslintrc`, `.prettierrc` (code quality)
- `tsconfig.json` (TypeScript)

### Constitutional element profile (actual analysis)

Applying v2 vocabulary to build-tooling:

#### Law 6 (Compiler ↔ Runtime Split) — STRONG

Build-tooling IS the compiler stage. Canonical Law 6
manifestation:
- Compiler stage: webpack + babel + WordPress plugins
  (build-time)
- Linker stage: WP enqueue system (request-time asset
  loading)
- Runtime stage: browser execution + WP block registration
  consumption

**Verdict**: Law 6 STRONG manifestation confirmed.

#### Law 3 (Authority Continuity) — PARTIAL

Block.json identity continuity from `src/block.json` →
`build/block.json` → server `register_block_type()`
consumption:
- Block name preserved across build pipeline
- Asset URL transformation (development paths → built paths)
  preserves logical identity

**Verdict**: Law 3 PARTIAL manifestation (less central than
in interactivity/i18n bounded contexts).

#### Law 1 (Declaration ≠ Exposure) — PARTIAL

Config declarations vs build outputs:
- `package.json` declares; `node_modules/` exposes
- `webpack.config.js` declares; `build/` exposes
- `src/block.json` declares; `build/block.json` exposes
- Multi-form gap (declared / built / loaded / registered)

**Verdict**: Law 1 PARTIAL multi-form gap (similar to
notices Phase 8.5+ chunk's 5-form gap, but at build pipeline
layer).

#### Federation Pattern — PARTIAL

Multiple plugins federate around shared wp-scripts:
- Plugin A uses wp-scripts; Plugin B uses wp-scripts;
  both inherit shared webpack/babel conventions
- Build outputs federated via WP enqueue registry
- Translation files federated via shared `wp i18n` tooling

**Verdict**: Federation PARTIAL manifestation.

#### Computational-architectural character — PARTIAL (DIFFERENT FORM)

Build-tooling exhibits compiler character BUT distinct from
style-engine + interactivity computational-architectural:
- style-engine: compiler ↔ browser CSS engine pipeline
- interactivity: server compiler ↔ JS reactive runtime
- **build-tooling: source compiler ↔ build artifacts**

Build-tooling computational character is **STATIC compilation**
(source → artifacts), NOT runtime-graph compilation. This is
structurally distinct.

**Verdict**: Computational-architectural PARTIAL but with
DIFFERENT character (static vs runtime-graph computational).

#### Doctrine 6 (Authority Access Mediation) — MOSTLY ABSENT

Build-tooling has minimal access gating:
- Minor: `npm install --save-dev` privilege requires
  filesystem write access (OS-level, not Doctrine 6)
- Build process doesn't gate WordPress authority access
- No capability checks, no routing gates, no signaling, no
  reactive subscriptions, no context reassignment

**Verdict**: Doctrine 6 MOSTLY ABSENT — absence prediction
B.1.b CONFIRMED.

#### Authority Mediation Surface intra-context density — ABSENT

No per-user gating in build process. Build process is
batch operation, not per-request mediation.

**Verdict**: ABSENT — absence prediction CONFIRMED.

#### Authority Interception Surface — ABSENT

No editor lifecycle interception. Build-tooling pre-dates
editor lifecycle (build → deploy → load → register →
editor).

**Verdict**: ABSENT — absence prediction CONFIRMED.

#### Law 3b 3b-react sub-character — ABSENT

Build-tooling is build-time, NOT reactive runtime. No
reactive subscription character. No PHP↔JS Bridge Pattern.

**Verdict**: ABSENT — absence prediction CONFIRMED.

#### Doctrine 5 paired operations — WEAK / ABSENT

Build pipeline is sequential (source → process → output),
NOT Arbitration ↔ Resolution paired operations character.

**Verdict**: ABSENT in pure form — absence prediction
CONFIRMED.

#### Bridge Pattern — ABSENT (build artifacts NOT cross-runtime bridges)

Build-tooling produces STATIC artifacts. Build artifacts
themselves are NOT PHP→HTML→JS bridges (those are runtime
bridges).

**Verdict**: ABSENT in build-tooling itself — absence
prediction CONFIRMED.
