# RULE — `selectors` (block.json) — style-engine attachment override surface

## WHEN

Defining a block in `block.json` and you need to **override the
default CSS selectors** that the style engine uses when generating
CSS rules for this block's styles. Use when:

- The block's wrapper is NOT the visual element to target (the
  styles need to apply to an inner element).
- Different block supports should target different elements
  (e.g., color on the wrapper, typography on an inner heading).
- Specific subfeatures need their own selectors to avoid CSS
  cascade issues (e.g., `text-decoration` resists override on a
  parent — needs targeted selector).

This is a **block.json field structurally** (declared at block
registration) but **semantically belongs to style-engine concerns**
(it's the API by which blocks tell the style engine WHERE to
attach generated CSS). Reading `selectors` as a "block.json minor
field" misses its depth — it is the **block-side declaration of
style-engine targeting overrides**.

This chunk is the **bridge from block-authoring closure into the
style-engine bounded context**. Style-engine is the runtime
authority concentrator that materializes schema declarations
(settings, styles, supports) into actual emitted CSS; `selectors`
is how blocks customize the engine's targeting for their styles.

## SHAPE

### 3-level selector hierarchy

| Level | Scope | Default behavior |
|---|---|---|
| **Root** | Block's primary CSS selector | `.wp-block-{namespace-and-slug}` (auto-generated from useBlockProps) |
| **Feature** | Per-block-support family (color, typography, border, etc.) | Falls back to root |
| **Subfeature** | Per-individual-style (background-color, text-decoration, etc.) | Falls back to feature root, then block root |

### Field metadata

| Property | Value |
|---|---|
| Type | `object` |
| Required | optional |
| Localized | no |
| Default | `{}` |
| Since | WP 6.3.0 |

### Root selector (the block's primary class)

```json
{
  "selectors": {
    "root": ".my-custom-block-selector"
  }
}
```

If unspecified, the root defaults to `.wp-block-{name}` (auto-
generated by `useBlockProps`). Override only when the block needs
a non-standard root class.

### Feature-level selectors (per block-support family)

```json
{
  "selectors": {
    "root":       ".my-custom-block-selector",
    "color":      ".my-custom-block-selector",
    "typography": ".my-custom-block-selector > h2"
  }
}
```

**Use case (per source):** colors apply to the wrapper, but
typography styles apply to an inner `<h2>` only. Without this
override, both would target the wrapper — the inner heading
wouldn't receive the typography styles.

### Subfeature-level selectors (per-individual-style)

```json
{
  "selectors": {
    "root":       ".my-custom-block-selector",
    "color":      ".my-custom-block-selector",
    "typography": {
      "root":            ".my-custom-block-selector > h2",
      "text-decoration": ".my-custom-block-selector > h2 span"
    }
  }
}
```

**Use case (per source):** `text-decoration` is hard to override
once applied to a parent. Targeted subfeature selector
(`> h2 span`) lets the style engine emit text-decoration on the
exact element that should receive it, avoiding cascade conflicts.

### Custom CSS selector (`selectors.css`)

Special-case selector field that maps to `styles.blocks.{name}.css`
in theme.json (NOT to a feature like color/typography):

```json
{
  "selectors": {
    "root": ".my-custom-block-selector",
    "css":  ".my-custom-block-selector > .inner-wrapper"
  }
}
```

OR object form with a `root` key:

```json
{
  "selectors": {
    "root": ".my-custom-block-selector",
    "css":  { "root": ".my-custom-block-selector > .inner-wrapper" }
  }
}
```

If unset, `selectors.css` falls back to the block's root selector.

### Shorthand (string) vs object form

For features, the value can be a **string** (shorthand: applies
to all subfeatures of that feature) OR an **object** with `root`
+ per-subfeature paths.

## REQUIRES

- **WP 6.3 or later** (per source: *"Since: WordPress 6.3.0"*).
- Each selector value MUST be a valid CSS selector
  (`document.querySelector` compatible).
- Selectors are **applied within the block's rendered DOM
  context** at output time — they target descendants of the
  block instance, not arbitrary site elements.
- Block MUST be registered server-side. Selectors are read by the
  style engine when generating CSS from theme.json styles —
  block.json must reach the PHP side for selectors to be
  consumed.
- Block's wrapper element MUST use `useBlockProps()` /
  `useBlockProps.save()` /
  `get_block_wrapper_attributes()` so the auto-generated
  `.wp-block-{name}` class exists for the default root selector.
- For `selectors.css` field: the corresponding theme.json path
  is `styles.blocks.{vendor/slug}.css` — that's where the actual
  custom CSS string lands. `selectors.css` only declares the
  TARGET selector, not the CSS content itself.

## INVARIANTS

- **Style-engine attachment override surface.** This field's
  semantics belong to the style-engine layer, even though it
  declared in block.json. Without a style engine that READS
  these declarations and uses them to compile per-block CSS,
  the field would have no effect. Mental model:
  - block.json declares: "for color styles on this block, use
    selector X; for typography, use selector Y"
  - style-engine reads: theme.json `styles.blocks.{name}.color
    .text` value + this block's `selectors.color` selector → emits
    `<X> { color: <value>; }`
- **3-level hierarchy with cascading fallback.** Resolution
  order for any `subfeature` (e.g., `color.background-color`):
  1. Check `selectors.{feature}.{subfeature}` — if set, use it.
  2. Else check `selectors.{feature}.root` — if set, use it.
  3. Else check `selectors.{feature}` (string form) — if set, use it.
  4. Else fall back to `selectors.root` — if set, use it.
  5. Else fall back to default `.wp-block-{name}` (auto from
     useBlockProps).
  This **fallback chain is unidirectional** (toward broader
  scope); siblings don't share unless explicitly configured.
- **`selectors.css` is structurally distinct.** Unlike
  feature-named keys (color, typography, border, etc.) that
  match block-support family names, `css` is a special key:
  - Targets the theme.json `styles.blocks.{vendor/slug}.css`
    custom-CSS field (NOT a block-support feature).
  - Falls back to block root if unset.
  - Accepts string OR object form with `root`.
  Treat `selectors.css` as the targeting hook for theme-author
  custom CSS, not as a block-support targeting.
- **Default root is `.wp-block-{namespace-and-slug}`.** The
  auto-class produced by `useBlockProps()`. Override only when
  the block needs a non-standard wrapper class — most blocks
  should accept the default. (This couples `selectors` to
  `block.wrapper-attributes` auto-class generation.)
- **Selectors are SERVER-SIDE consumed.** Block.json's selectors
  field is read by the style engine running in PHP at render
  time (or at theme.json compilation time). The block's
  declaration is processed by the engine to emit CSS rules.
  This is why server-side block registration is required.
- **Subfeature selectors solve specific CSS cascade problems.**
  Source identifies `text-decoration` as a canonical problem
  case: text-decoration is hard to override once set on an
  ancestor. Subfeature selectors let the engine emit
  text-decoration ONLY on the element that should have it, not
  on a parent that would force the style downward.
- **Shorthand vs object form is per-feature decision.** Within
  a single `selectors` block, color may be string-form
  (`"color": ".x"`) while typography is object-form
  (`"typography": { "root": ".x", "text-decoration": ".y" }`).
  Mix freely based on whether per-subfeature granularity is
  needed.
- **Feature names match block-support family names.** Documented
  examples cover `color`, `typography`. Any block support
  family (border, spacing, dimensions, filter, etc.) presumably
  uses the same key naming. ⚠ Exact list of valid feature keys
  is not enumerated in source; safest assumption is "every
  block support family name plus `css` plus `root`".
- **`selectors.filter.duotone` is documented elsewhere.** The
  `block.supports.filter` chunk shows
  `selectors.filter.duotone: ".wp-block-image img"` — this is
  the same selectors API at work, targeting filter.duotone's
  realization to descendant `<img>` elements rather than the
  wrapper. Consistent with the 3-level hierarchy
  (`filter.duotone` is a subfeature path).
- **Inheritance model is feature-isolated.** Setting
  `selectors.color: ".x"` does NOT affect
  `selectors.typography` resolution. Each feature's resolution
  is independent.
- **No "global override" mechanism via selectors.** This API
  is per-block-type only — declared in block.json. There is
  no theme-wide way to override block selectors (themes that
  need different selectors for a block must work through
  styles.blocks.{name}.css escape hatch with manual
  selectors).
- **Style engine is the actual consumer.** The selectors field
  is meaningless without a style engine that reads it. This
  chunk's full ontology requires understanding the style engine
  itself (see RELATED for forthcoming chunks).
- ⚠ **Minimum WP version: 6.3** for the selectors field as a
  whole. Specific feature key support (e.g., `selectors.filter`)
  may have arrived in different versions. Frontmatter `wp_min`
  set to 6.3.

## ANTIPATTERNS

- ❌ Setting `selectors.root` to a selector that doesn't actually
  match the block's wrapper at render time. The style engine
  emits CSS targeting the declared selector; if no element
  matches, styles don't apply.
- ❌ Forgetting `useBlockProps()` while declaring custom
  selectors. The default root selector relies on the
  auto-generated class; without useBlockProps, that class
  doesn't exist on the wrapper.
- ❌ Using selectors that reach OUTSIDE the block's DOM subtree
  (e.g., `body .my-block`). Selectors should be self-contained
  to the block's instance — leaking outside affects unrelated
  parts of the page.
- ❌ Confusing `selectors.css` with `styles.css` in theme.json.
  - `selectors.css` (this field) declares the SELECTOR for
    where custom CSS lands.
  - `styles.blocks.{name}.css` (theme.json) declares the actual
    CSS CONTENT.
  Both work together: theme provides CSS, block tells engine
  where to attach it.
- ❌ Setting subfeature selectors that don't match block.json
  feature/subfeature naming. Subfeature names follow CSS
  property names (e.g., `background-color`, `text-decoration`,
  `font-size`); using arbitrary names produces no effect.
- ❌ Trying to use `selectors` to override theme-side selectors
  for OTHER blocks. The field is per-block-type — only affects
  the block declaring it. To affect other blocks, theme must
  use their respective `selectors` (if those blocks declare
  them) or use `styles.blocks.{name}.css` escape hatch.
- ❌ Hardcoding selectors that include `:hover`, `:focus`, or
  other pseudo-classes within `selectors.{feature}`. The style
  engine emits property-value pairs; pseudo-class scoping
  belongs in `styles.css` (escape hatch), not in selectors
  declaration.
- ❌ Declaring `selectors` on WP < 6.3. Field is silently
  ignored on older versions; fallback is always the default
  root selector. Feature-detect or set `wp_min: "6.3"`.
- ❌ Treating `selectors` as documentation. The selectors are
  CONSUMED by the engine at render time; they MUST match the
  actual rendered DOM structure or styles fail silently.

## RELATED

- `block.json-supports-field` — block-support family names
  (color, typography, border, spacing, dimensions, filter,
  etc.) are the **feature key namespace** for selectors.
  selectors customize WHERE generated styles for these
  supports get attached.
- `block.wrapper-attributes` — `useBlockProps` /
  `get_block_wrapper_attributes` produces the default
  `.wp-block-{name}` root selector. selectors.root overrides
  what the engine targets, but doesn't change what
  useBlockProps produces on the actual element. Both must
  align (or selectors.root must specify a different element
  the block actually renders).
- `theme-config.json-styles-color` — color realizations consume
  `selectors.color` (or fall back to `selectors.root`). Same
  pattern for all per-capability styles.
- `theme-config.json-styles-typography` — typography realizations
  consume `selectors.typography`. Object form
  (`{ root, text-decoration, ... }`) supports per-subfeature
  targeting.
- `theme-config.json-styles-css` — `styles.blocks.{name}.css`
  custom-CSS realization is targeted by `selectors.css` in this
  field. The two layers connect: theme.json provides CSS
  content, block.json provides the selector for where it lands.
- `block.supports.filter` — `selectors.filter.duotone` is
  documented in the supports.filter chunk; same selectors API
  at work for filter realization targeting.
- `style-engine.css-emission` (cross-context, planned) — the
  style engine is the ACTUAL CONSUMER of these declarations.
  Future chunk on the engine's compilation passes will explain
  how selectors translate into emitted CSS rules. selectors is
  meaningless without the engine.
