# RULE — `supports` field — block capability declaration

## WHEN

Defining a block in `block.json` and you want the editor to expose
**common feature controls** (color picker, typography panel, spacing
controls, alignment, anchor, etc.) without writing the controls
yourself. The `supports` field is the gateway to capability flags
provided by core; opting in registers attributes, generates UI, and
auto-applies the resulting CSS class names / inline styles to the
block wrapper.

This chunk covers the **`supports` field as a mechanism**. Each
individual flag (`color`, `typography`, `spacing`, etc.) gets its own
chunk because each has distinct editor effects, serialization output,
and theme.json interaction.

## SHAPE

```json
{
  "supports": {
    "color": { "background": true, "text": true },
    "spacing": { "padding": true, "margin": [ "top", "bottom" ] },
    "typography": { "fontSize": true, "lineHeight": true },
    "anchor": true,
    "align": [ "wide", "full" ]
  }
}
```

Field metadata:

| Property | Value |
|---|---|
| Type | `object` |
| Required | optional |
| Localized | no |
| Default | `{}` (no supports opted in) |

Each top-level key inside `supports` is a **capability flag**. Values
range from `boolean` (simple on/off) through `string[]` (subset
selection) to nested `object` (fine-grained sub-flags per capability).
The exact value shape depends on the flag — see per-flag chunks.

### Companion APIs to render the generated attributes

The capability declaration alone does NOT apply styling. The
generated attributes flow to the block wrapper via:

JS (static / interactive blocks):
```js
import { useBlockProps } from '@wordpress/block-editor';

function BlockEdit() {
  const blockProps = useBlockProps();
  return <div { ...blockProps }>Hello World!</div>;
}

function BlockSave() {
  const blockProps = useBlockProps.save();
  return <div { ...blockProps }>Hello World!</div>;
}
```

PHP (dynamic blocks via `render_callback` or `render`):
```php
function render_block() {
  $wrapper_attributes = get_block_wrapper_attributes();
  return sprintf( '<div %1$s>%2$s</div>', $wrapper_attributes, 'Hello World!' );
}
```

## REQUIRES

- The block MUST be registered on the **server** (PHP) for capability
  flags that depend on attribute persistence, theme.json interaction, or
  block hooks. Client-only registration loses these effects.
- The block's `Edit` and `save` (or PHP render) MUST spread
  `useBlockProps()` / `useBlockProps.save()` / `get_block_wrapper_attributes()`
  onto the block's outer element. Skipping this step means the editor
  controls work in the inspector but the resulting classes/inline styles
  never reach the rendered markup.
- Some flags require `apiVersion: 2` or higher (e.g., features that
  depend on iframed-editor or wrapper-class injection patterns).
  Setting `apiVersion: 3` in `block.json-basic-metadata` is the safe
  default.
- Theme-side opt-in via `theme.json` is required to make some
  flag-generated controls actually expose presets (e.g., `color`
  controls show no palette unless `settings.color.palette` is defined).

## INVARIANTS

Opting into a `supports` flag triggers a **cascade of effects**, not a
single behavior change. Standard capability-semantics sub-sections:

### Editor effects

- A control panel appears in the block inspector (e.g., Color panel,
  Typography panel) — auto-generated by core based on the flag(s)
  declared.

### Attribute effects

- The flag implicitly extends the block's `attributes` schema. For
  example, `supports.align: true` adds an `align` attribute (string,
  no default). The block's own `attributes` field can declare a
  default (`attributes.align.default = "right"`) but the schema
  entry comes from the flag.

### Wrapper effects

- The user's selection materializes as one or more of: CSS class names
  (e.g., `has-accent-4-color`, `has-background`), inline `style`
  properties (e.g., `style="color: #...; padding: 16px;"`), or `id`
  attributes (for `anchor`). All injected via `useBlockProps` /
  `get_block_wrapper_attributes`.
- If you spread `useBlockProps` but ALSO set conflicting attributes
  (e.g., your own `className` prop after spreading), the spread's
  values may be overridden — order matters.

### Serialization effects

- The selected values are serialized into the block delimiter
  (`<!-- wp:my-plugin/foo {"backgroundColor":"contrast"} -->`) so they
  round-trip across save/load.

### theme.json interaction

- Capability flags that take preset values (color, typography,
  spacing) reference `theme.json` presets via
  `var(--wp--preset--{slug})` CSS custom properties. The editor's
  preset picker shows whatever is declared in
  `settings.{capability}.{collection}` in theme.json.

### General invariants

- The flag itself does not write CSS — it injects classes/inline
  styles that downstream stylesheets (core, theme, or your own)
  must define. Some classes (`has-{slug}-color`) come from core's
  preset class generation; others depend on theme stylesheet rules.
- ⚠ **Minimum WP version varies per flag.** Many flags were introduced
  in different WP releases (e.g., `background` in WP 6.4+,
  `viewStyle` in WP 6.5+). The `supports` field itself is original
  Block API, but specific flags' availability is per-flag. See the
  per-flag chunks for accurate `wp_min` values.

## ANTIPATTERNS

- ❌ Recreating controls that `supports` already provides. If you
  write your own color picker when `supports.color.background = true`
  would do it, you double the UI, fork the persistence path, and
  break theme.json preset integration.
- ❌ Spreading `useBlockProps()` without applying it to the outermost
  element of the block, or splitting attributes across multiple
  wrappers. The generated classes/styles MUST land on the same
  element the user perceives as "the block".
- ❌ Forgetting `get_block_wrapper_attributes()` in dynamic
  `render_callback` / `render.php`. Editor controls will appear and
  attributes will persist, but the front-end rendered HTML won't
  reflect them. Silent disconnect.
- ❌ Declaring `supports.color: { background: true }` without
  ensuring the theme provides a color palette. The control appears
  but the picker is empty unless `theme.json` `settings.color.palette`
  is populated (or the user provides a custom hex if `custom: true`
  is also set).
- ❌ Treating `supports` as schema-only metadata. It's not just
  declarative — it actively rewrites the block's attribute schema,
  generates controls, mutates wrapper output, and consumes theme.json.
  The flag is a 5-layer cascade.
- ❌ Setting `supports.align = true` (boolean) when only specific
  alignments make sense. Use the array form (`["left", "right",
  "full"]`) to scope which alignments are exposed.

## RELATED

- `block.json-basic-metadata` — `apiVersion` gates flag availability.
- `block.json-attributes` — attribute schema; supports flags
  implicitly extend this schema.
- `block.json-assets` — `style` / `editorStyle` define the CSS rules
  that consume the classes generated by supports.
- `block.wrapper-attributes` — `useBlockProps` /
  `get_block_wrapper_attributes()` semantics; the receiver of all
  supports-injected output.
- `block.dynamic-rendering` — how `$attributes` reach the render
  function; where capabilities materialize into HTML.
- **Per-flag rules (planned):** `block.supports.color`,
  `block.supports.typography`, `block.supports.spacing`,
  `block.supports.border`, `block.supports.align`,
  `block.supports.anchor`, `block.supports.dimensions`, etc.
- `theme-config.json-presets` (cross-context) — the theme.json
  presets that supports-color / typography / spacing controls
  consume.
