# RULE — Register all plugin blocks via metadata collection

## WHEN

- Multi-block plugin (2+ blocks).
- Targeting WordPress 6.8 or later.
- All blocks register with the same default args (no per-block
  customization needed at registration time — if needed, see
  `block.register-collection-with-args`).
- Build pipeline can produce a `blocks-manifest.php` file.

## SHAPE

```php
function my_plugin_register_blocks() {
    wp_register_block_types_from_metadata_collection(
        __DIR__ . '/build',
        __DIR__ . '/build/blocks-manifest.php'
    );
}
add_action( 'init', 'my_plugin_register_blocks' );
```

Function signature:

| Parameter | Type | Required | Notes |
|---|---|---|---|
| `$path` | `string` | yes | Absolute path to the directory containing the built blocks (typically `build/`). |
| `$manifest` | `string` | yes | Absolute path to the `blocks-manifest.php` file (typically inside the same `build/`). |

Single function call registers every block whose `block.json` was rolled
into the manifest at build time.

## REQUIRES

- WordPress 6.8 or later.
- `blocks-manifest.php` exists at the manifest path. Generated by
  `wp-scripts build-blocks-manifest`, or by passing `--blocks-manifest`
  to `wp-scripts build` / `wp-scripts start`.
- Each block's `block.json` lives under the build directory at the path
  recorded in the manifest.
- Hook on `init` action.

## INVARIANTS

- The manifest is **build-time output** — must be regenerated whenever
  any `block.json` is added/removed/changed. Stale manifest = stale
  registration.
- Block `name` values across all manifested blocks MUST be globally
  unique (same constraint as individual `register_block_type()` calls).
- If a block needs different args than the manifest defaults at
  registration time, this rule does not fit — switch to
  `block.register-collection-with-args`.

## ANTIPATTERNS

- ❌ Calling this function on WP < 6.8. Function does not exist; will
  throw `Call to undefined function`. Use feature detection
  (`function_exists()`) or fall back to `register_block_type()` per
  block when version-targeting older WP.
- ❌ Hand-writing `blocks-manifest.php`. Manifest format is build-tool
  contract; manual edits get overwritten on next build and risk schema
  drift.
- ❌ Registering individual blocks with `register_block_type()` after
  this call (unless those blocks are intentionally outside the
  manifest). Causes duplicate registration warnings.
- ❌ Pointing `$path` to `src/` instead of `build/`. The manifest
  references built block.json paths; mismatch breaks lookup.

## RELATED

- `block.register-via-block-json` — fallback for single-block plugins or
  WP < 6.7 environments.
- `block.register-collection-with-args` — WP 6.7+ variant when
  per-block `$args` (e.g., different `render_callback`) are required.
- `block.json-schema` — block.json field reference.
- `tooling.wp-scripts-build-blocks-manifest` — build command that
  produces the manifest file consumed here.
