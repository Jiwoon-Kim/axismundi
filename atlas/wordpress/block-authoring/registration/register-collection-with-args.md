# RULE — Register manifest, then register each block with custom `$args`

## WHEN

- Multi-block plugin where some blocks need different `$args` at
  registration time (e.g., distinct `render_callback` per block).
- Targeting WordPress 6.7 or later (but you can't use the WP 6.8+
  one-call API because of the per-block args requirement).
- Build pipeline produces a `blocks-manifest.php`.

## SHAPE

```php
function my_plugin_register_blocks() {
    wp_register_block_metadata_collection(
        __DIR__ . '/build',
        __DIR__ . '/build/blocks-manifest.php'
    );

    register_block_type( __DIR__ . '/build/block-one' );

    register_block_type( __DIR__ . '/build/block-two', array(
        'render_callback' => 'my_plugin_render_block_two',
    ) );
}
add_action( 'init', 'my_plugin_register_blocks' );
```

Function signature for `wp_register_block_metadata_collection()`:

| Parameter | Type | Required | Notes |
|---|---|---|---|
| `$path` | `string` | yes | Absolute path to the build directory containing the blocks. |
| `$manifest` | `string` | yes | Absolute path to `blocks-manifest.php`. |

Pattern: register the manifest first (so subsequent `register_block_type()`
calls read metadata from memory, not disk), then call
`register_block_type()` for each block with whatever `$args` it needs.

## REQUIRES

- WordPress 6.7 or later.
- `blocks-manifest.php` exists at the manifest path. Generated by
  `wp-scripts build-blocks-manifest` or `wp-scripts build/start
  --blocks-manifest`.
- Each subsequent `register_block_type()` path is one of the blocks the
  manifest knows about (path matches a manifest entry).
- All calls hooked on `init`.

## INVARIANTS

- Manifest registration MUST come before any `register_block_type()`
  calls in the same hook invocation. Reverse order means the per-block
  calls fall back to disk reads, defeating the optimization.
- Each `register_block_type()` call still reads from the in-memory
  manifest if the path matches; it does NOT re-read block.json from disk.
- If you call `register_block_type()` with a path NOT in the manifest,
  WordPress falls back to disk read for that block (silent — no warning).

## ANTIPATTERNS

- ❌ Using this rule when no `$args` differ across blocks. Use
  `block.register-collection-php` (WP 6.8+) instead — single function
  call, less code.
- ❌ Calling on WP < 6.7. `wp_register_block_metadata_collection()`
  does not exist; use individual `register_block_type()` calls per
  block.
- ❌ Hand-writing `blocks-manifest.php`. Build-tool contract; manual
  edits get overwritten and risk drift.
- ❌ Registering the manifest twice (e.g. on a re-entrant hook).
  Behavior is undefined; treat as one-shot per request lifecycle.
