# RULE — `add_rewrite_rule` and `WP_Rewrite` — ordered regex pattern arbitration for URL-to-query-var resolution

## WHEN

You need to reason about how WordPress turns a
URL request into the query variables that drive
template selection and content rendering. Use
this knowledge when:

- Adding a custom permalink pattern for a plugin
  (e.g., `/products/{slug}/{tab}/`).
- Diagnosing "my custom URL returns 404" — almost
  always: rule not registered, registered after
  parse_request fired, flushing not done after
  registration, or earlier rule shadowing yours.
- Choosing between `add_rewrite_rule` (a single
  pattern → query template), `add_rewrite_endpoint`
  (URL-fragment endpoints), and
  `add_rewrite_tag` (permalink structure tokens).
- Understanding why core post types' permalinks
  *just work* — they're built from rewrite rules
  generated by `register_post_type`.
- Reading core's `WP::parse_request` and
  following the rule-walking loop that resolves
  the request.

This chunk does **not** cover:

- The template hierarchy that *follows* rewrite
  resolution (covered in
  `site-building.template-hierarchy-and-resolution`).
  Rewrite rules choose **query vars**; template
  hierarchy chooses **which file renders** for
  the resolved query.
- The `WP_Query` mechanics that consume the
  resolved query vars. Rewrite resolution
  produces the input to `WP_Query`; the query
  itself is its own topic.
- The full permalink-structure customization
  surface (`%postname%`, custom date formats,
  etc.). This chunk uses permalink structure as
  context but doesn't catalog it.
- `add_rewrite_endpoint` and `add_rewrite_tag`
  in detail — Section F covers them as adjacent
  surfaces; their full behavior is its own
  topic.

The principle this chunk operates under: **A
rewrite rule is one of many ordered regex
candidates against an incoming URL. The matching
walk is **first-match-wins, terminal, and
discarding** — once a rule matches, its query
template determines the resolved query vars and
**no later rule is evaluated**. This is one of
the cleanest *positive* Law 4 (Arbitration
Compiler) instances in WordPress: ordered
candidates, terminal selection, real
discarding.**

## SHAPE

### A. The rewrite rules registry

WordPress stores all registered rewrite rules in
a single option (`rewrite_rules`) compiled by
`WP_Rewrite`. The shape of the option:

```
[
    'pattern_1_regex' => 'index.php?vars=…',
    'pattern_2_regex' => 'index.php?vars=…',
    'pattern_3_regex' => 'index.php?vars=…',
    …
]
```

Each entry maps a **regex pattern** (matched
against the request URI, minus the home URL
prefix) to a **query var template** (a query
string of `index.php?key=value` pairs, where
values can reference regex capture groups via
`$matches[N]`).

The order of entries in the array is the order
in which `WP::parse_request` evaluates them —
top to bottom — against the request URI. The
first entry whose regex matches *wins*; its
query template determines the resolved query
vars; later entries are skipped.

The order is built from:

- `'top'`-positioned `add_rewrite_rule` calls
  (in registration order, before defaults).
- WordPress's default rules (generated from the
  active permalink structure, registered post
  types, taxonomies, etc.).
- `'bottom'`-positioned `add_rewrite_rule` calls
  (in registration order, after defaults).

Three buckets, concatenated. Within each bucket,
registration order is preserved.

### B. Rule registration

```php
add_rewrite_rule(
    string $regex,                  // the pattern to match
    string $query,                  // query template with $matches[N] references
    string $position = 'top'        // 'top' or 'bottom'
);
```

A typical registration:

```php
add_action( 'init', function() {
    add_rewrite_rule(
        '^products/([^/]+)/?$',
        'index.php?post_type=product&product=$matches[1]',
        'top'
    );
} );
```

The example registers: when a URL matches
`/products/{anything-non-slash}/`, the
single-product page for that slug renders.

Three properties to pin:

- **`'top'` vs `'bottom'`** is about *whether
  this rule should beat the defaults*. A
  custom rule for `/products/(.+)/` placed at
  `'bottom'` would lose to any default rule
  that matches first (e.g., page-name lookup).
  `'top'` ensures custom logic precedes
  defaults; `'bottom'` is for fallbacks.
- **Registration alone is not enough.** The
  rule lives in PHP memory until
  `flush_rewrite_rules()` writes it to the
  option (Section C). Without flushing, the
  rule won't appear in the next request's
  `rewrite_rules` lookup.
- **The query template is a string with regex
  back-references.** `$matches[1]`,
  `$matches[2]`, etc. interpolate at
  resolution time. This is the bridge between
  the regex's captured groups and the
  resolved query vars.

### C. The flush mechanism

The rewrite rules array is *cached in the
`rewrite_rules` option*. Adding rules at runtime
without flushing means they're available to
that PHP process but not persisted to the
option — and `WP::parse_request` reads from the
option, not from runtime registrations.

`flush_rewrite_rules()` rebuilds the option:

```php
flush_rewrite_rules();  // expensive: regenerates and writes the option
```

Two patterns for when to flush:

- **On plugin activation.** If a plugin
  registers rewrite rules, flush in the
  activation hook so the option is written
  once at install time:

  ```php
  register_activation_hook( __FILE__, function() {
      // Re-register rules so they're current.
      myplugin_add_rules();
      flush_rewrite_rules();
  } );
  ```

- **On rule changes.** If rules change between
  versions or based on configuration, flush at
  the moment of change. *Don't* flush on every
  request — the option write is expensive.

Forgetting to flush is the dominant "my
rewrite rule isn't working" failure. The rule
exists in PHP, but the option doesn't include
it; `parse_request` doesn't see it; the URL
404s.

### D. Request parsing — the first-match-wins walk

`WP::parse_request()` is where the arbitration
runs. Simplified pseudocode:

```
1. Determine request URI (strip home URL prefix).
2. Load the rewrite_rules option.
3. For each (regex, query_template) entry in the array, in order:
       if (preg_match(regex, request_uri, $matches)):
           # MATCH FOUND.
           Parse query_template, interpolating $matches[N].
           Set query vars on $this (the WP instance).
           BREAK out of the loop.
4. (If no match found: fall through to default
   query handling, which usually 404s for
   unknown URLs.)
5. Continue request lifecycle (load template, etc.).
```

Three properties of this walk to pin:

- **The walk is **ordered**.** Entries are
  evaluated in array order. The order matters
  decisively for which rule wins.
- **The walk is **terminal**.** Once a match
  is found, the loop breaks. Later rules are
  never evaluated for this request.
- **Other rules are **discarded** for this
  request.** Not "deferred"; not "considered
  with lower priority." Simply not evaluated.
  Their existence has no effect on the
  outcome.

The combination — *ordered + terminal +
discarding* — is exactly the canonical Law 4
shape. This is not a queue (queue processes
all). This is not a layered merge (merge
combines all). This is not federation (federation
runs all callbacks). This is **first-match-wins
selection from ordered candidates**, with later
candidates structurally inert once a winner
emerges.

### E. Built-in rule generators

Most rewrite rules in a typical site come not
from `add_rewrite_rule` but from generators that
WordPress runs automatically:

| Source                          | Rules generated                                                  |
| ------------------------------- | ---------------------------------------------------------------- |
| Permalink structure             | Patterns for posts, dates, categories, authors, paginated archives |
| Registered custom post types    | Single-item URLs, archive URLs, paged variants                   |
| Registered custom taxonomies    | Term archive URLs, paged variants                                |
| Endpoints                       | URL fragments like `/feed/`, `/page/N/` (Section F)              |

The total can run into hundreds of rules on a
site with several custom post types and
taxonomies. The walk in Section D processes all
of them per request — in order — until a match.

The arbitration shape is the same regardless of
where the rules came from. A custom rule
registered via `add_rewrite_rule('top')`
competes with the same evaluation logic as a
core-generated rule for `/category/{slug}/`.

### F. Adjacent registration surfaces

Three related but distinct registration
mechanisms:

**`add_rewrite_endpoint`** — adds URL fragments:

```php
add_rewrite_endpoint( 'json', EP_PERMALINK );
// Now /post-slug/json/ resolves with the 'json' query var set.
```

Endpoints modify the *query var generation* by
adding extension fragments to existing URL
patterns. They don't add separate rules in the
`add_rewrite_rule` sense — they extend the
query template that matched rules produce.

**`add_rewrite_tag`** — adds permalink
structure tokens:

```php
add_rewrite_tag( '%color%', '([^&]+)', 'color=' );
// Now permalinks can include %color%, generating rules accordingly.
```

Tags are even lower-level — they extend the
*permalink structure language* itself, allowing
new tokens. Permalink-structure-driven rule
generation will then produce rules using these
tokens.

**Direct rule writing** (`add_rewrite_rule`) is
the right tool when:

- The pattern doesn't fit existing permalink
  structure semantics.
- You want a one-off URL-to-query-var
  mapping.
- The query template needs query vars that
  endpoints / tags don't naturally produce.

The three exist as a small DSL family for
URL-shape customization. This chunk's principal
mechanism — ordered regex arbitration — applies
once any of them have produced their resulting
rules in the rewrite array.

## WHY

### Why first-match-wins rather than all-rules-evaluated

Each request resolves to *one* set of query
vars. Multiple matches would require the system
to merge or arbitrate the conflicting outputs
— which is exactly what first-match-wins is
designed to avoid. The first match's query
template wins; later matches don't add
contributions because there's no meaningful way
to combine them (which post_type wins if two
rules disagree?).

The cost is rule order matters absolutely —
plugins must coordinate or accept the order
they get. The benefit is the resolution is
deterministic, predictable, and inexpensive.

### Why the rules are stored as an option

`WP::parse_request` runs on every front-end
request. Re-computing the rewrite rules from
scratch (re-running every plugin's registration,
every permalink-structure generation, every
post-type / taxonomy generator) on every
request would be expensive.

The option caches the compiled rules. The
trade-off is the **flush requirement** — when
rules change, the option must be regenerated.
Plugin developers carry the burden of flushing
at registration-change moments.

### Why `'top'` vs `'bottom'` rather than numeric priority

The rewrite rules table is finite (typically
hundreds of entries) and the bucket model is
sufficient — most custom rules are either
"override defaults" (`top`) or "fallback after
defaults" (`bottom`). A numeric priority would
allow finer ordering but wouldn't change the
fundamental first-match-wins semantics; it
would just add complexity to the registration
API.

The two-bucket model gives plugins one
intentional choice (override-or-fallback) and
defers further ordering to registration order
within the bucket.

### Why this is structurally different from hook priority or layer merge

These look adjacent but the mechanisms differ:

- **Hook priority** orders callbacks; *all
  callbacks fire*; outputs compose.
  → Composition.
- **Layer merge** orders contributions; *all
  layers contribute*; outputs combine.
  → Aggregation.
- **Rewrite rules** order patterns; *one
  pattern wins*; others are discarded.
  → Arbitration.

The structural difference is whether **all
participants execute** or whether **execution
terminates at first success**. Hook and merge
are the former; rewrite is the latter.

This is the Law 4 boundary: arbitration
exists when the mechanism *terminates with one
winner and discards the rest*. Composition,
aggregation, federation, parameterization,
ordering-as-display all involve order without
that termination.

## WHEN NOT

Skip rewrite rules if:

- **Default WordPress URLs work.** Posts,
  pages, categories, custom post types with
  default rewrite settings — all of these are
  generated rules; you don't need `add_rewrite_rule`.
- **You can use endpoints instead.** If the
  custom URL fragment can attach to existing
  URL patterns (e.g., `/post-slug/json/`),
  `add_rewrite_endpoint` is simpler than rolling
  a custom rule.
- **Pretty permalinks are off.** If the site's
  permalink structure is set to "Plain" (the
  ?p=N format), rewrite rules don't apply —
  WordPress reads query vars from the URL
  directly. Custom rules silently do nothing.
- **The "URL" is for AJAX or REST.** Use
  `register_rest_route` for REST API endpoints;
  the rewrite rules path is for content URLs,
  not API endpoints.

## COUNTER-PATTERNS

### Anti-pattern 1 — Registering without flushing

```php
add_action( 'init', function() {
    add_rewrite_rule( '^my/(.+)/?$', 'index.php?my_var=$matches[1]', 'top' );
} );
```

…and never calling `flush_rewrite_rules()`.
The rule lives in PHP memory each request but
is never written to the option. `parse_request`
reads the option; sees no matching rule; URL
404s.

Flush at activation:

```php
register_activation_hook( __FILE__, function() {
    add_rewrite_rule( '^my/(.+)/?$', 'index.php?my_var=$matches[1]', 'top' );
    flush_rewrite_rules();
} );
```

…or after configuration changes that affect
rules.

### Anti-pattern 2 — Flushing on every request

```php
add_action( 'init', function() {
    add_rewrite_rule( '^my/(.+)/?$', '...', 'top' );
    flush_rewrite_rules();  // every request — expensive
} );
```

Flushing rebuilds and writes the option. Doing
this on every request adds a database write
per request — a meaningful tax on every page
load. Flush only when rules actually change.

### Anti-pattern 3 — Registering at `'bottom'` and expecting precedence

```php
add_rewrite_rule( '^products/(.+)/?$', '...', 'bottom' );
```

`'bottom'` rules lose to defaults. If a default
rule matches `/products/widget-x/` first
(e.g., a page named "products"), the custom
rule never gets evaluated. Use `'top'` for
"this should win over defaults"; use
`'bottom'` for "consider this only if defaults
didn't match."

### Anti-pattern 4 — Forgetting to add custom query vars

```php
add_rewrite_rule( '^my/(.+)/?$', 'index.php?my_var=$matches[1]', 'top' );
flush_rewrite_rules();
// Then: var is set in URL but $wp->query_vars doesn't recognize my_var
```

Custom query vars in the rewrite output need
registration via the `query_vars` filter:

```php
add_filter( 'query_vars', function( $vars ) {
    $vars[] = 'my_var';
    return $vars;
} );
```

Without this, WordPress strips `my_var` from
parsed query vars; the custom logic doesn't
see the value.

### Anti-pattern 5 — Confusing rule registration with rule activation

```php
// Plugin A and Plugin B both register:
add_rewrite_rule( '^products/(.+)/?$', '...', 'top' );  // Plugin A
add_rewrite_rule( '^products/(.+)/?$', '...', 'top' );  // Plugin B
```

The second registration *also runs*, but the
first one to be registered (in walk order) wins
when a URL matches both. Plugin B's registration
exists but is structurally inert — its regex
will never be evaluated for that URL because
Plugin A's matches first.

This is the *terminal selection* property: even
identical registrations matter only for the
first one in the walk order. Plugin B's rule
participates in the storage but not in the
outcome.

### Anti-pattern 6 — Treating the rewrite rules table as inspectable user-facing config

```php
// Some plugin offering a "URL Rules" admin page:
$rules = get_option( 'rewrite_rules' );
foreach ( $rules as $regex => $query ) {
    echo '<li>' . esc_html( $regex ) . ' → ' . esc_html( $query ) . '</li>';
}
```

This works mechanically but exposes hundreds
of internal regex patterns to a non-technical
user. The rewrite rules table is a *compiled
internal substrate*, not user-facing
configuration. If users need URL customization,
build a higher-level abstraction (permalink
structure UI, custom-pattern templates) that
generates rules behind the scenes.

## OPERATIONAL NOTES

The rewrite-rules substrate's interpretive
shape, in proportional v2 vocabulary:

- **Law 4 (Arbitration Compiler)** is the
  central fit, in a *positive control* form
  that the chunk explicitly tests for. The
  rewrite rules walk is **the canonical
  example of arbitration in WordPress's PHP
  layer**:
  - Multiple candidates exist (the
    `rewrite_rules` array).
  - Walk is ordered (array order, top →
    bottom).
  - First match wins (`break` on regex
    success).
  - Selection determines outcome (matched
    rule's query template runs).
  - Discarded candidates are **structurally
    inert for this request** (later rules not
    evaluated).
  This is the *positive Law 4 anchor* the
  recent KB asked for. The 8-member anti-Law-4
  inventory documents mechanisms that *look*
  arbitration-shaped without being so. The
  rewrite-rules walk is the inverse: a
  mechanism that *is* arbitration in the
  literal mechanism-matching sense, regardless
  of surface vocabulary. Naming Law 4 here is
  not just clarifying — it is **doctrinally
  load-bearing**: without positive examples,
  Law 4 risks becoming a category that exists
  only as the negative pole of repeated
  non-fits. Rewrite rules confirm Law 4's
  positive applicability outside the previously-
  established anchors (template hierarchy,
  `locate_template`, navigation fallback).
- **Law 1 (Declaration ≠ Exposure)** is
  **PRIMARY** alongside Law 4 (rare to have
  both at this strength). The rule is
  *declared* via registration; it is *exposed*
  (matched, used to set query vars) only when
  its regex matches a URL, AND when no earlier
  rule matched first. Most registered rules
  are *never* exposed in any given request.
  Naming Law 1 here is genuinely clarifying
  because the *gap* between "this rule is
  registered in the option" and "this rule
  governed this request" is the structural
  defining property — most rules sit
  perpetually in the option without ever
  governing a request.
- **Doctrine 5 (Authority Continuity)**
  appears *lightly*. The regex pattern + query
  template tuple is the identity surface
  across registration, storage, evaluation,
  and the resulting query vars. Worth one
  mention; not a section.

What this chunk is **not** about:

- **Federation.** *Important non-fit, with
  precision.* Multiple plugins register rules
  into the shared `rewrite_rules` table, which
  *looks* federation-shaped — many participants,
  one registry. But the rendering side
  fundamentally differs:
  - **Federation** (hooks, wp-data-registry,
    SlotFill) means *all participants execute*
    on dispatch.
  - **Rewrite rules** mean *one participant
    executes; others discarded* on each
    request.
  Open registration with first-match-wins
  selection is *registration federation +
  resolution arbitration* — a hybrid that is
  *not pure federation*. The substrate shape
  matters more than the registration shape:
  what the framework does *after* the
  contributions are gathered determines whether
  it's federation. Rewrite rules: doesn't
  qualify as federation under the definition
  the KB has been using.
- **Law 3b (Cross-Runtime Authority Continuity
  Bridge).** All resolution happens in the
  PHP request runtime. No cross-runtime
  authority preservation. Omitted.
- **Doctrine 6 (Authority Mediation).** No
  access mediation. Rewrite rules don't
  enforce capabilities. Any code can register
  any rule. Omitted.
- **Law 6 (Compiler ↔ Runtime Split).** The
  compiled rewrite_rules option is generated
  at flush time and consumed at runtime —
  but this is *cache pre-computation*, not a
  build / runtime split in the framework's
  sense. The flush happens within PHP, not in
  a separate build pipeline. Omitted.
- **Section X archetypes.** A pattern-arbitration
  substrate is not a "civilization." Same
  framework-omission discipline as the
  surrounding chunks. Omitted.

## CHECKLIST

When using `add_rewrite_rule`:

- [ ] Register on `init` (or earlier as
      appropriate); don't register inside
      request-time hooks like `parse_request`.
- [ ] Use `'top'` to override defaults; use
      `'bottom'` for fallbacks. Don't expect
      `'bottom'` to ever fire if defaults
      cover the URL space.
- [ ] Register custom query vars via the
      `query_vars` filter; otherwise the rule
      sets a var that WordPress strips out.
- [ ] Flush rules at the right time: in
      activation hooks, after configuration
      changes — but **not** on every request.
- [ ] When registering a custom post type or
      taxonomy with `rewrite => array(...)`,
      know that core generates rules for you;
      explicit `add_rewrite_rule` is for
      cases beyond what type/taxonomy
      registration covers.
- [ ] Don't surface the raw `rewrite_rules`
      option to end users. It's an internal
      substrate; build higher-level UIs over
      it.
- [ ] When debugging "URL 404s," check first
      whether the rule is in the
      `rewrite_rules` option (flush?), then
      whether it's positioned correctly
      (`'top'` vs `'bottom'`), then whether
      its query vars are registered.
