# RULE — `styles.spacing` — generated-token realization layer

## WHEN

Configuring a theme's `theme.json` `styles.spacing.*` to set the
**actual spacing values** (block-gap, margin, padding) realized in
CSS. This is the **third realization-layer chunk** in KB and
completes the styles subtype taxonomy:

| styles chunk | Subtype | Pattern |
|---|---|---|
| `styles.color` | Token consumption | preset → CSS variable → property |
| `styles.typography` | Computational realization | preset → clamp / responsive → property |
| **`styles.spacing`** | **Generated-token realization** | **spacingScale-generated preset → property + blockGap variable propagation** |

This chunk also closes **the most cross-coupled capability in KB —
`blockGap`** — by completing its full-stack journey across 6 layers
(see invariants).

## SHAPE

### 3 fields with TWO distinct value forms

| Field | Type | CSS effect |
|---|---|---|
| `blockGap` | `string \| { ref }` | Sets `--wp--style--block-gap` CSS custom property (gated by `settings.spacing.blockGap`). |
| `margin` | `{ top, right, bottom, left }` | Per-side margin styles. **Object structure — first non-`string|{ref}` form in styles.*.** |
| `padding` | `{ top, right, bottom, left }` | Per-side padding styles. Same object structure as margin. |

⚠ Value form for individual sides (top/right/bottom/left) is
not explicitly typed in source. Inferred from pattern: each side
likely accepts `string | { ref }` (matching the family grammar).
Verification needed.

### blockGap example (string + preset reference)

```json
{
  "styles": {
    "spacing": {
      "blockGap": "var:preset|spacing|40"
    }
  }
}
```

This:
1. Resolves `var:preset|spacing|40` to the value declared in
   `settings.spacing.spacingSizes` (or generated by spacingScale).
2. Emits as `--wp--style--block-gap: <resolved>`.
3. Consumed by layout's flex/grid `gap` CSS at runtime.

### margin / padding examples

```json
{
  "styles": {
    "spacing": {
      "margin": {
        "top":    "var:preset|spacing|60",
        "bottom": "var:preset|spacing|60"
      },
      "padding": {
        "top":    "var:preset|spacing|40",
        "right":  "var:preset|spacing|40",
        "bottom": "var:preset|spacing|40",
        "left":   "var:preset|spacing|40"
      }
    }
  }
}
```

⚠ Whether shorthand string form (`"padding": "16px"`) is also
accepted is not explicitly documented. Standard pattern in source
shows object form only.

### Per-block-type and per-element scoping

```json
{
  "styles": {
    "spacing": { "blockGap": "var:preset|spacing|40" },
    "blocks": {
      "core/group": {
        "spacing": {
          "padding": { "top": "var:preset|spacing|60" }
        }
      }
    }
  }
}
```

Same 4-level cascade as styles.typography:
1. Block instance `style.spacing.*` in delimiter (highest).
2. `styles.blocks.{name}.spacing.*`.
3. `styles.elements.{name}.spacing.*` (rarely used for spacing).
4. Top-level `styles.spacing.*` → body (lowest).

## REQUIRES

- Setting MUST be declared under `theme.json` `styles.spacing`,
  `styles.elements.{name}.spacing`, OR
  `styles.blocks.{name}.spacing`.
- For **`blockGap` to take effect**, `settings.spacing.blockGap`
  MUST be `true`. Source: *"Sets the `--wp--style--block-gap` CSS
  custom property when settings.spacing.blockGap is true."*
  Without the settings gate, the value is declared but the CSS
  variable does NOT generate — silent no-op.
- For preset references (`"var:preset|spacing|{slug}"`):
  the slug MUST match an entry in `settings.spacing.spacingSizes`
  OR a slug auto-generated by `settings.spacing.spacingScale`.
- For literal string values: MUST be valid CSS length values
  with units (`"16px"`, `"1rem"`, `"clamp(...)"`, etc.).
  Bare numbers are invalid.
- Per-side object form (margin/padding): each declared side
  (top/right/bottom/left) takes the same value grammar
  (`string | { ref }`). Omitted sides do not emit (selectors
  retain their inherited / default values for those sides).
- For blockGap to PROPAGATE to children: the parent block must
  use a layout type that consumes the gap (flex / grid / flow).
  See `block.supports.layout` for layout type semantics.

## INVARIANTS

- **Generated-token realization subtype** — confirmed third
  styles taxonomy member. spacingScale's algorithmic generation
  in settings produces a preset list; styles consumes those
  presets via `var:preset|spacing|{slug}` references. Compare:
  - color: token CONSUMPTION (lookup, not computed).
  - typography: COMPUTATIONAL realization (clamp synthesis applies
    to preset values themselves at the CSS variable level).
  - spacing: GENERATED-token realization (presets are pre-computed
    by spacingScale, then consumed in styles like color presets).
  Spacing combines color's consumption pattern with settings-side
  generation — split between layers.
- **`blockGap` is the first "full-stack capability" in KB.**
  After this chunk, blockGap appears across **6 layers** —
  the most cross-coupled capability documented:

  | Layer | blockGap role |
  |---|---|
  | `block.supports.spacing.blockGap` | per-block opt-in (declares the control) |
  | `appearanceTools` (mediation) | bundled (1 of 3 spacing gates in bundle) |
  | `settings.spacing.blockGap` (gate) | theme gates whether `--wp--style--block-gap` is generated |
  | **`styles.spacing.blockGap` (this chunk)** | provides the VALUE assigned to the CSS variable |
  | `block.supports.layout` (consumer) | layout type consumes `var(--wp--style--block-gap)` as `gap` CSS |
  | (style-engine, future) | actual `--wp--style--block-gap` emission + per-block-context resolution |
- **Cross-layer dependency documented IN the styles field
  description.** Source for `blockGap`:
  *"Sets the `--wp--style--block-gap` CSS custom property when
  settings.spacing.blockGap is true."*
  This is **unprecedented** — most settings/styles fields don't
  reference their counterpart in the source description. Spacing's
  cross-layer coupling is so tight that the documentation surfaces
  it explicitly. Treat this as a SIGNAL that blockGap deserves
  cross-layer reasoning, not isolated per-layer treatment.
- **Two distinct value-form patterns within one chunk.**
  - `blockGap` uses `string | { ref }` (matching styles.color,
    styles.typography pattern).
  - `margin` / `padding` use `{ top, right, bottom, left }`
    object form — **first non-string|{ref} structural pattern in
    the styles.* layer**.
  This reflects CSS-layer reality: margin/padding semantically
  decompose into 4 sides (longhand), while blockGap is a
  single-value concept (gap takes one length).
- **Per-side decomposition is a longhand realization choice.**
  `margin: { top: "16px" }` realizes as CSS `margin-top: 16px`
  (longhand), not as `margin: 16px 0 0 0` (shorthand). This:
  - Allows partial overrides (only `top` declared, others
    inherited).
  - Avoids the CSS shorthand reset behavior (where shorthand
    resets unspecified longhand values).
- **NO preset registries in styles.spacing.** spacingSizes and
  spacingScale live in `settings.spacing` only. styles consumes
  the resulting preset slugs via `var:preset|spacing|{slug}`
  references — same pattern as styles.color consuming
  `settings.color.palette` slugs.
- **Asymmetry profile — middle ground (predicted and confirmed):**
  - color: settings 14 / styles 3 → **4.7x** (BROAD settings,
    NARROW styles, gap filled by elements path)
  - typography: settings 16 / styles 12 → **1.3x** (near 1:1
    mirror of gates ↔ values)
  - **spacing: settings 8 / styles 3 → 2.7x** (settings is
    GENERATOR, styles is CONSUMER — distinct roles, not mirror)
  Three distinct asymmetry profiles for three capabilities. Each
  reflects different settings/styles role-division.
- **`styles.elements.*` rarely used for spacing.** Unlike
  typography (heavy element specialization for headings / links /
  buttons), spacing rarely needs per-element variation. Element
  scoping exists but is not the primary path. Block-type
  specialization (`styles.blocks.{name}.spacing.*`) is more
  common.
- **Vertical pipeline (with generative source):**
  ```
  settings.spacing.spacingScale       (algorithm declaration)
      ↓ at theme load
  preset values synthesized            (e.g., 7 sizes)
      ↓ each emits as
  --wp--preset--spacing--{slug}        (CSS custom property)
      ↓ referenced in
  styles.spacing.padding.top = "var:preset|spacing|40"
      ↓ resolves to
  padding-top: var(--wp--preset--spacing--40)
      ↓ at runtime
  resolved spacing value applied
  ```
  Compare blockGap pipeline:
  ```
  styles.spacing.blockGap = "var:preset|spacing|40"
      ↓ if settings.spacing.blockGap is true
  --wp--style--block-gap: var(--wp--preset--spacing--40)
      ↓ consumed by
  layout flex/grid gap CSS rule
      ↓
  child blocks separated by resolved gap value
  ```
- **`{ ref }` form for cross-spacing references.** Same caveat
  as styles.color and styles.typography. Likely use:
  inheriting margin from another scope. Verification-needed.
- **Settings.spacing.units relationship is implicit.** The
  documented `settings.spacing.units` array
  (`["px","em","rem","vh","vw","%"]`) governs what units users
  can input via inspector controls. Whether values declared in
  styles.spacing.* are also constrained by this list is
  unverified — likely the units list is editor-side governance
  only, with theme.json styles able to use any valid CSS unit.
- **NO `inset` / shorthand convenience aliases.** Unlike CSS's
  `inset` shorthand (top/right/bottom/left in one), the
  styles.spacing per-side object lists each side individually.
  No documented shortcut for "all sides equal" in object form;
  shorthand string form acceptance unverified.
- ⚠ **Minimum WP version unknown.** Spacing styles are part of
  theme.json since early versions; per-side object form likely
  matches the original API. Frontmatter `wp_min` is
  `"verification-needed"`.

## ANTIPATTERNS

- ❌ Setting `styles.spacing.blockGap` without enabling
  `settings.spacing.blockGap: true`. The value is silently
  ignored — `--wp--style--block-gap` is not generated. Always
  pair the two layers.
- ❌ Setting `styles.spacing.padding: "16px"` (string shorthand)
  expecting it to apply to all 4 sides. Source documents object
  form `{ top, right, bottom, left }` only; shorthand acceptance
  is unverified. Use object form for safety.
- ❌ Hardcoding spacing values in CSS instead of declaring via
  styles. Bypasses the spacing scale cascade — preset changes
  in settings don't propagate; per-block overrides have no
  effect.
- ❌ Looking for `styles.spacing.spacingSizes` or
  `spacingScale`. Registry declarations live in settings,
  NOT styles. Realization layer doesn't declare token
  registries.
- ❌ Setting `blockGap` on a block that doesn't use a layout
  type. The CSS variable generates but no `gap` rule consumes
  it. blockGap requires a layout (flex / grid / flow) to be
  visible. See `block.supports.layout`.
- ❌ Using per-element `styles.elements.h1.spacing.padding`
  expecting heading-specific padding. Element scoping for
  spacing is uncommon and may have limited support; per-block
  scoping (`styles.blocks.core/heading.spacing`) is more
  reliable.
- ❌ Declaring `styles.spacing.margin: { top: "primary" }`
  (bare slug). Must be full preset reference:
  `"var:preset|spacing|primary"`. Bare slugs don't resolve.
- ❌ Renaming spacing preset slugs in settings after styles
  references them. References break silently — CSS variable
  resolves to undefined.
- ❌ Mixing units inconsistently within the same scope (e.g.,
  `padding.top: "16px"`, `padding.bottom: "1rem"`). Allowed
  but produces inconsistent vertical rhythm. Use the spacing
  scale (preset references) for consistency.
- ❌ Treating `blockGap` as equivalent to `margin` between
  siblings. blockGap is INTERNALLY emitted by the parent's
  layout style; margin is OUTWARDLY emitted by each child.
  Different ownership → different runtime behavior (margin
  collapsing, etc.).

## RELATED

- `theme-config.json-settings-spacing` — the registry +
  generative substrate this realizes. Settings declares
  spacingSizes / spacingScale (token generation); styles
  consumes the resulting preset slugs via
  `var:preset|spacing|{slug}` references. Vertical pipeline
  closure: settings (generation) → CSS variable (synthesized
  preset) → styles (consumption) → wrapper / element selectors
  → final CSS.
- `theme-config.json-settings-layout` — `blockGap` value
  emitted here as `--wp--style--block-gap` is CONSUMED by
  layout's flex/grid `gap` CSS at runtime. Cross-layer
  coupling: spacing styles produce the value; layout styles
  consume it.
- `block.supports.spacing` — per-block opt-in surface that
  pairs with this realization. supports declares whether
  inspector controls appear for users to override these
  theme-level styles per instance.
- `block.supports.layout` — consumes `--wp--style--block-gap`
  CSS variable in flex/grid gap properties. blockGap is
  meaningful only when a parent block has a layout type.
- `block.markup-representation` — per-instance
  `style.spacing.*` in block delimiter is the cascade leaf
  (highest specificity). Theme styles set defaults; per-instance
  values override.
- `theme-config.json-styles-color` — adjacent realization with
  **token consumption** pattern. Compare: color (pure lookup)
  vs spacing (lookup of generated tokens) vs typography
  (lookup of computationally-resolved tokens).
- `theme-config.json-styles-typography` — adjacent realization
  with **computational realization** pattern (continuous
  interpolation via clamp). spacing is the **discrete**
  computational realization counterpart (multiplicative scale
  generation in settings → static preset values consumed in
  styles).
