# Note federation route contract (increment 4)

Status: **#4a and #4b implemented for the 0.0.1 release.** Increment 3 owns the storage substrate;
increment 3.5 owns language inheritance. This document freezes the HTTP,
source, projection, and HTML-view boundaries across the two increment 4 slices.

Increment 4 was implemented as two reviewable commits and is released only after both
are complete:

- **#4a - Federation readiness + source + JSON-LD**: freeze the authored identity
  fields at first federation exposure, resolve the canonical Note URI to a Note-owned
  source, and project Note or Tombstone JSON-LD with exact 200/404/410 semantics.
- **#4b - HTML view-model + Note fallback template**: give the canonical human `url`
  a real representation while establishing the common object view-model contract.

JSON-LD is not released without #4b because the renderer requires a meaningful human
`url`, and FEP-b2b8 recommends an HTML representation Link.

## Layering invariant

Object Projections (OP) is a base plugin; Note is a product plugin that already
depends on OP. **OP never reads `wp_ax_notes` and never depends on Note.** Note
registers its request-source resolver, transformer, and view-model adapter through OP
extension seams. OP calls those callbacks without knowing Note storage.

## First federation exposure

A public route is federation exposure even when no Create Activity has yet been
delivered. Identity fields therefore cannot wait until increment 5 to become stable.

On the first transition of an `ax_note` post to `publish`, a Note-owned preparation
hook performs these writes before the object becomes publicly readable:

1. Resolve language through `axismundi_note_effective_language()` and store the
   normalized BCP-47 value in `language_tag`.
2. Refresh the unlocked envelope attribution from the current `post_author`, verify
   that it is a public, local, handle-locked Actor, but do not lock it yet.
3. Strictly resolve the current explicit-plus-body mention set. This validation does
   not copy body-derived mentions into the envelope; it only prevents first exposure
   with an invalid recipient.
4. After every prior check succeeds, freeze attribution through the Note-owned lock
   API. `attribution_locked_at` is the final readiness marker and is never set for a
   partially prepared Note.

Preparation fails closed. A post may remain locally published, but it is not a
federatable source until both snapshots exist. Existing published Notes that predate
the hook also remain 404 until an explicit preparation/republish path succeeds. The
transformer and both HTTP routes are read-only and never perform either write.

Increment 5 consumes these already-frozen fields when recording Create; it does not
mint a second attribution or language snapshot.

## OP request source seam (#4a)

`axismundi_op_current_source()` keeps its existing precedence:

1. resolved Actor;
2. queried Core `WP_Post`;
3. a preserve-by-default product filter.

Its contract is:

```php
apply_filters( 'axismundi_op_current_source', $source )
    -> source | WP_Error( 'ax_op_not_found' ) | null
```

- A resolver preserves a non-null source unless the request explicitly names its own
  product namespace. This narrow override is necessary because WordPress may ignore an
  unregistered query and resolve a static front-page `WP_Post`.
- A replacement source means that one product plugin claimed and resolved its exact
  namespace.
- `ax_op_not_found` means a product namespace was present but malformed, aliased, or
  unknown. Negotiated JSON-LD and the #4b HTML route map it to 404.
- `null` means no product claimed the request; WordPress continues normally.

This distinction prevents `/?ax_note=bad` from falling through to a homepage 200.
Callers used only for discovery headers or `<link rel="alternate">` ignore errors and
emit no metadata.

## Note request identity (#4a)

The Note resolver claims a request whenever the `ax_note` key is present. It resolves
only the exact canonical path and UUID generated by
`axismundi_note_object_uri($uuid)`.

The optional `activitypub` representation selector may accompany the canonical query
for consistency with OP negotiation, but it is never part of object identity. Any
other extra key, duplicated key, alternate path, userinfo, or fragment is an
`ax_op_not_found` result. An unknown but syntactically valid UUID is also 404.

The Note CPT remains private (`publicly_queryable=false`, no Core query var), so this
resolver and the #4b HTML handler are the only owners of the namespace.

## Note-owned source value (#4a)

The source is not a bare `WP_Post`. It contains:

- the immutable envelope row;
- an optional `WP_Post`;
- the canonical object URI.

An active source requires its Post. A tombstone remains resolvable from the envelope
after permanent Post deletion. The source value is Note-owned; OP treats it as opaque.

## Resolution and disclosure matrix (#4a)

An **active federatable Note** satisfies every condition below:

- envelope `object_status=active`;
- matching `WP_Post` exists, has type `ax_note`, status `publish`, and no password;
- envelope attribution is locked and identifies a local, public, handle-locked Actor;
- stored `language_tag` is a normalized non-`und` BCP-47 snapshot;
- visibility and mention input can be resolved by the shared Activities audience
  resolver.

| Source state | Requester | Outcome |
|--------------|-----------|---------|
| active federatable, public / unlisted | anyone | `200` Note |
| active federatable, followers / mentioned | anonymous | `404` |
| active but any readiness condition fails | anyone | `404` |
| tombstone envelope, Post optional | anyone | `410` Tombstone |
| claimed malformed/aliased/unknown Note URI | anyone | `404` |
| no Note namespace | anyone | existing source is preserved; OP adds no handling |

Authenticated viewing of followers/mentioned Notes is deferred until a separate
authorization contract. UUID opacity is not an authorization mechanism.

## Generic OP Tombstone contract (#4a)

Tombstone behavior is an OP-wide representation rule, not a Note special case:

- `axismundi_op_finalize_object()` accepts a Tombstone with privacy-minimal required
  members (`id`, `type`) instead of requiring `attributedTo` and `url`.
- `formerType` and `deleted` are emitted when known; content, summary, audience,
  attribution, and human URL are not copied from the deleted object.
- The generic object router maps a finalized `type: Tombstone` to `410 Gone` and adds
  `Cache-Control: no-store`; other successful objects remain 200.
- The HTML view-model uses the same object status and also returns 410.

These rules are reusable by future local object products and align with the existing
privacy-minimal QuoteAuthorization tombstone route.

## Note transformer purity (#4a)

The Note transformer performs no DB write and no network request. For an active source
it emits:

- `type: Note`, canonical `id`, frozen `attributedTo`, and a text/html Link-valued
  `url` pointing to the same negotiated canonical URI;
- `content` plus `contentMap` keyed by the **stored language snapshot** only;
- `to` / `cc` from the shared Activities audience resolver over stored visibility and
  the current explicit-plus-body mention set;
- resolvable Mention tags with fully-qualified names. A stale Actor descriptor may be
  omitted during live projection, while increment 5 keeps strict recipient validation
  at the Create/Update authoring boundary;
- optional `inReplyTo`, `context`, `sensitive`, content warning, `published`, and
  `updated` members.

For a tombstone source it emits only the privacy-minimal Tombstone fields. It never
reads live language inheritance and never reconstructs deleted content.

## HTML view-model and template boundary (#4b)

OP owns a storage-neutral `object_view_model` adapter registry and the dynamic
`axismundi/object-view` block. Note owns the canonical functional single-object
block template; a theme or user-saved template may override its slug. Product
plugins provide adapters and templates while storage remains separate.

The routed surface introduced in increment 4 is the canonical Note HTML URL:

- active public/unlisted -> 200 fallback object view;
- unauthorized followers/mentioned or unready/unknown -> 404;
- tombstone -> 410 privacy-minimal view.

This increment **does not replace the active WordPress theme's existing single-post
Article routing or template**. A Core Article adapter may expose the same view-model to
blocks or future fallback use, but OP does not seize `single-post`. A remote-cache
adapter may share the model and sanitization contract, while a new public local route
for cached remote objects remains a separate routing decision.

All rendered cached HTML uses the existing OP positive allowlist and never causes a
passive remote media request.

Like and Announce controls appear only when their existing capability checks pass.
Quote capability remains false until outbound Quote authoring is implemented. No
interaction control is rendered for Tombstones or unauthorized objects.

## Increment 5 lifecycle

Increment 5 records a complete finalized Note Object snapshot as the embedded `object`
of Create and Update. Activities compares that snapshot with the immediately preceding
lifecycle event: duplicate callbacks converge, a changed representation records Update,
and returning to an older representation remains a new Update. Create after Delete begins
a new generation.

The Gutenberg REST write is marked request-locally so `save_post` cannot record the
pre-envelope intermediate state; `rest_after_insert_ax_note` records only after the
structured field commits. Invalid recipients return a REST error and no Activity. The
stored Actor and language snapshots are immutable after first exposure.

Withdrawal and permanent deletion record a URI-only Delete using the preceding ledger
audience, never the now-withdrawn Object. Hard deletion tombstones first and aborts until
that Delete is durable. Note writes no transport state; Bridge independently queues each
committed outbound Activity.

- A public local route for cached remote objects, if separately approved.
